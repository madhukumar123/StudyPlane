Let us take a **single concrete problem** and solve it using four paradigms, then analyze what happens at the memory and CPU level.

---

# 🧩 Problem Statement

**Given a list of integers, compute the sum of squares of even numbers.**

Example:

Input:

```
[1, 2, 3, 4, 5]
```

Output:

```
2² + 4² = 4 + 16 = 20
```

---

# 1️⃣ Imperative Style

Example: C

```c
int sum_of_squares(int arr[], int n) {
    int sum = 0;
    for(int i = 0; i < n; i++) {
        if(arr[i] % 2 == 0) {
            sum += arr[i] * arr[i];
        }
    }
    return sum;
}
```

### Hardware-Level Behavior

Memory:

* `arr[]` contiguous in memory
* `sum` stored in register or stack
* No heap allocation

CPU:

* Loop counter increment
* Branch check (`if`)
* Multiplication
* Addition

Characteristics:

* Very predictable branch pattern
* Excellent cache locality
* Minimal abstraction overhead

---

# 2️⃣ Object-Oriented Style

Example: Java

```java
class NumberProcessor {
    public int sumOfSquares(List<Integer> list) {
        int sum = 0;
        for (Integer num : list) {
            if (num % 2 == 0) {
                sum += num * num;
            }
        }
        return sum;
    }
}
```

### Hardware-Level Behavior

Memory:

* `List` object
* `Integer` objects (boxed values)
* Heap allocations for each element

Extra overhead:

* Pointer dereferencing
* Object headers
* Possibly virtual method calls

CPU:

* Load object reference
* Dereference pointer
* Unbox integer
* Arithmetic operations

Compared to C:

* More indirection
* More cache misses possible
* GC involvement

---

# 3️⃣ Functional Style

Example: Python

```python
def sum_of_squares(nums):
    return sum(x*x for x in nums if x % 2 == 0)
```

Or more purely functional:

```python
from functools import reduce

def sum_of_squares(nums):
    return reduce(lambda acc, x: acc + x*x,
                  filter(lambda x: x % 2 == 0, nums),
                  0)
```

### Hardware-Level Behavior

Memory:

* Generator object created
* Lambda objects created
* Each integer is a Python object
* Reference counting involved

CPU:

* Function calls for `lambda`
* Iterator protocol calls (`__iter__`, `__next__`)
* Dynamic type checks
* Bytecode interpretation

Overhead:

* Dynamic dispatch
* Heap allocations
* Reference counting increments/decrements

But:

* Code is highly expressive
* Easy to parallelize conceptually

---

# 4️⃣ Declarative Style

Example: SQL

Assume a table:

```sql
SELECT SUM(value * value)
FROM numbers
WHERE value % 2 = 0;
```

### Hardware-Level Behavior

You do not control the loop.

Internally the database engine:

* Parses query
* Builds execution plan
* Possibly uses index
* Vectorized execution
* May parallelize automatically

Memory:

* Buffer pools
* Query plan structures
* Possibly SIMD vector operations

CPU:

* Optimized loops in C
* Possibly parallel threads
* Highly tuned cache-friendly algorithms

Declarative style often outperforms hand-written loops when data is large because of query optimizer intelligence.

---

# Side-by-Side Structural Comparison

| Paradigm    | Who Controls Loop? | Memory Allocation | Indirection | Optimization Control |
| ----------- | ------------------ | ----------------- | ----------- | -------------------- |
| Imperative  | You                | Minimal           | Low         | Full                 |
| OOP         | You                | Moderate          | Medium      | Moderate             |
| Functional  | Runtime            | High              | Medium–High | Low                  |
| Declarative | Engine             | Engine-managed    | Hidden      | Engine decides       |

---

# Deeper CPU Insight

### Imperative

```
LOAD
COMPARE
JUMP
MULTIPLY
ADD
```

Straight-line predictable execution.

---

### OOP

```
LOAD pointer
LOAD object field
LOAD vtable (if virtual)
CALL method
```

Extra memory fetches.

---

### Functional

```
CALL lambda
RETURN
CALL next()
TYPE CHECK
REF COUNT UPDATE
```

Frequent runtime overhead.

---

### Declarative

Engine may use:

* Vectorized instructions
* SIMD
* Multi-threading
* Cache-aware block processing

Often fastest for large-scale data.

---

# Engineering Takeaway

All four paradigms compute the same mathematical result.

But at hardware level they differ in:

* Memory layout
* Allocation behavior
* Branch predictability
* Indirection layers
* Optimization authority

---

# My Professional Opinion

For performance-critical core loops:

* Imperative is safest.

For maintainable enterprise systems:

* OOP hybridized with functional concepts works best.

For large data processing:

* Declarative systems (databases, Spark-like engines) are superior.

For concurrency-safe systems:

* Functional style reduces shared-state bugs.
