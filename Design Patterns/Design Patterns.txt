I will structure **Design Patterns** as an **architectural learning hierarchy**, similar to the C++ and DSA breakdowns.
Design patterns are not merely coding tricks; they are **reusable architectural solutions to recurring design problems**.

I will classify them into **conceptual layers → classical patterns → advanced patterns → architectural patterns → modern system patterns**.

---

# PHASE 1 — Foundations of Software Design

Before learning patterns, you must understand **why patterns exist**.

## 1. Software Design Principles

These principles give birth to most patterns.

### SOLID Principles

* **Single Responsibility Principle (SRP)**
  A class should have only one reason to change.

* **Open Closed Principle (OCP)**
  Software entities should be open for extension but closed for modification.

* **Liskov Substitution Principle (LSP)**
  Derived classes must be substitutable for base classes.

* **Interface Segregation Principle (ISP)**
  Clients should not depend on interfaces they do not use.

* **Dependency Inversion Principle (DIP)**
  High-level modules should not depend on low-level modules.

---

## 2. Other Important Design Principles

* DRY (Don't Repeat Yourself)
* KISS (Keep It Simple)
* YAGNI (You Aren't Gonna Need It)
* Separation of Concerns
* Composition over inheritance
* Encapsulation

These principles **drive the creation of patterns**.

   ↓

# PHASE 3 — Classic GoF Design Patterns

The canonical patterns defined in **Design Patterns: Elements of Reusable Object-Oriented Software**.

They are divided into **three categories**.

---

# 1. Creational Patterns

These control **object creation mechanisms**.

## Singleton

Ensures only one instance exists.

Applications:

* Logging system
* Configuration manager

---

## Factory Method

Delegates object creation to subclasses.

Use when:

* Object type depends on runtime conditions.

---

## Abstract Factory

Creates families of related objects.

Example:

GUI frameworks creating:

```
Windows UI
Linux UI
Mac UI
```

---

## Builder

Separates object construction from representation.

Example:

* Complex objects like documents or HTTP requests.

---

## Prototype

Creates objects by cloning existing ones.

Useful when object creation is expensive.

---

# 2. Structural Patterns

These focus on **object composition**.

## Adapter

Converts one interface into another.

Example:

* Legacy API integration.

---

## Bridge

Separates abstraction from implementation.

Used when systems have **multiple dimensions of variation**.

---

## Composite

Represents part-whole hierarchies.

Example:

```
File system
Directory
File
```

---

## Decorator

Adds behavior dynamically.

Example:

* Middleware
* Logging wrappers

---

## Facade

Provides simplified interface to complex subsystems.

Example:

* Database access layers.

---

## Flyweight

Reduces memory usage by sharing objects.

Example:

* Text editors storing characters.

---

## Proxy

Acts as a placeholder for another object.

Examples:

* Lazy loading
* Access control

---

# 3. Behavioral Patterns

These manage **communication between objects**.

---

## Strategy

Encapsulates interchangeable algorithms.

Example:

```
Payment Strategy
CreditCard
UPI
NetBanking
```

---

## Observer

Defines publish–subscribe relationship.

Example:

* Event systems
* UI frameworks

---

## Command

Encapsulates requests as objects.

Example:

* Undo / redo systems.

---

## Iterator

Sequentially access elements of a collection.

---

## Mediator

Centralizes communication between components.

Example:

* Chat room server.

---

## Memento

Captures object state for restoration.

Example:

* Undo operations.

---

## State

Changes object behavior depending on state.

Example:

* Network connection states.

---

## Template Method

Defines skeleton of algorithm.

Subclasses override certain steps.

---

## Visitor

Separates operations from object structure.

Used in:

* Compilers
* AST processing.

---

# PHASE 4 — Advanced Design Patterns

These extend classical patterns for complex systems.

---

## Dependency Injection

Objects receive dependencies instead of creating them.

Benefits:

* Loose coupling
* Testability

---

## Service Locator

Central registry for dependencies.

---

## Null Object Pattern

Avoids null checks by using a default object.

---

## Lazy Initialization

Object created only when needed.

---

## Object Pool

Reuse expensive objects.

Example:

* Database connection pools.

---

# PHASE 5 — Concurrency Design Patterns

Important in modern systems.

---

## Thread Pool

Manages worker threads efficiently.

---

## Producer–Consumer

Decouples production and processing.

---

## Reactor Pattern

Handles asynchronous I/O events.

Used in:

* Web servers
* Networking frameworks

---

## Active Object Pattern

Separates method invocation from execution.

---

# PHASE 6 — Architectural Patterns

These operate at **system-level architecture**, not just classes.

---

## MVC (Model View Controller)

Separates:

```
Data
UI
Logic
```

Used in web frameworks.

---

## Layered Architecture

Typical enterprise architecture:

```
Presentation Layer
Business Logic Layer
Data Access Layer
Database
```

---

## Microservices Architecture

System split into independent services.

Benefits:

* Scalability
* Fault isolation

---

## Event Driven Architecture

System reacts to events.

Used in:

* Streaming systems
* Distributed systems

---

## Hexagonal Architecture

Also called **Ports and Adapters**.

Goal:

Decouple business logic from external systems.

---

# Architectural Need for Design Patterns

Patterns exist because software systems must solve recurring problems.

### Key problems solved:

1. **Code reuse**
2. **Loose coupling**
3. **Maintainability**
4. **Scalability**
5. **Flexibility**

Without patterns, systems become:

```
Tightly coupled
Hard to modify
Hard to test
```
